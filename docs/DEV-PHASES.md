# 开发阶段规划（落地版）

本文将 PRD / TECH 中的能力拆分为可交付的开发阶段，方便后续前后端 **按模块并行** 开发。

## 阶段 0：基础框架搭建（已完成）

- **后端**
  - Maven 多模块结构（common/model/dao/ai/service/api/job）
  - MySQL / Redis / MinIO / ES 等依赖在 `TECH.md` 与 `sql/`、`deploy/` 中约定
  - MyBatis-Plus：分页插件、逻辑删除配置、基础 Mapper / Service 模板
  - Spring Security：JWT 骨架、双域路由：`/api/app/**`、`/api/admin/**`、`/api/common/**`
  - SpringDoc OpenAPI：`common` / `app` / `admin` 三个分组 + Bearer 鉴权方案
  - 全局配置：CORS、Jackson 时区与时间格式、统一异常处理、请求 traceId 过滤器
  - SQL：`schema.sql` / `seed.sql`，与 `catering-model` 实体一一对应
  - ES：信息帖索引 mapping 与创建脚本（`deploy/es`）
- **前端**
  - 管理端：`catering-admin`（Vue 3 + Vite + TypeScript + Element Plus + Pinia + Vue Router）
    - 代理 `/api` → `http://localhost:8080`
    - 占位路由：登录页、后台布局、工作台，健康检查调用后端 `/api/common/health`
  - 用户端：`catering-app`（UniApp Vue3 H5）
    - 基础首页 + `/api/common/health` 调用封装
    - Vite 代理 `/api` → `http://localhost:8080`

> 通过 `mvn test`、前端 `npm run build` 完成基础冒烟验证。

## 阶段 1：账号与地区（MVP 先决条件）

**状态：已实现（v0.2）**

**目标**：让用户端和管理端都能“登录 + 选地区”，为后续发布/审核打基础。

- **后端**
  - 地区 API（`/api/common/regions/**`）：
    - 用户端：按层级获取省/市/区县树（仅启用的地区）
    - 管理端：增删改查、启用/停用、排序
  - 用户端登录方案（v1）：
    - 推荐：手机号 + 验证码（短信对接可后置，先 mock 通路）
    - 登录成功签发 **APP_USER** 类型 JWT
  - 管理端登录：
    - 管理员账号密码登录，签发 **ADMIN_USER** 类型 JWT，携带角色
  - 使用 `sys_config` 读取默认有效期、续期时长、日发布上限
- **前端**
  - 管理端：
    - 完成登录页表单与登录 API 对接（保存 Token + 路由守卫）
    - 地区管理页面（城市/区县 CRUD）
  - 用户端：
    - 登录入口（手机号登录 or 临时登录按钮）+ Token 存储
    - 城市/区县选择组件（使用地区 API 数据）

## 阶段 2：信息帖发布与审核闭环

**目标**：实现从“发布表单 → 人工审核 → 信息上架”的最小闭环，优先支持招聘与转让。

- **后端**
  - 用户端发布草稿/提交接口：
    - 至少实现招聘、转让两类信息的必要字段校验
    - 状态流转：草稿 → 待审核
  - 图片上传：MinIO 上传凭证接口（`/api/common/upload-token`），用户端使用直传
  - 审核接口（管理端）：
    - 待审列表、详情
    - 通过、驳回（原因模板 + 自定义说明）
    - 写入 `post_audit_record`
  - 审核通过后：
    - 状态改为 `APPROVED`，设置 `expire_at`
    - 后续阶段再接 ES 写入
- **前端**
  - 管理端：
    - 审核队列列表页
    - 审核详情页（字段 + 图片 + 历史违规次数）
  - 用户端：
    - 发布表单（两类）：字段校验 + 图片上传
    - 我的发布列表（草稿、待审核、已通过、已驳回、已过期）

## 阶段 3：列表 / 筛选 / 详情 / 电话展示

**目标**：让浏览者能“找信息 + 看详情 + 登录后拨号”，覆盖五类信息。

- **后端**
  - 列表接口（按类型）：
    - 分页、按发布时间排序
    - 支持城市/区县与各类型专属筛选（见 PRD 第 4 章）
    - 仅返回 `APPROVED` 且未过期信息
  - 详情接口：
    - 控制电话展示逻辑（未登录/已登录/已过期/已下架）
  - 免费置顶：
    - 管理端设置 `is_top/top_until`
    - 列表排序时优先置顶，过期自动失效
- **前端**
  - 用户端：
    - 列表页 + 筛选条件组件
    - 详情页：电话显示规则 + 拨号按钮
  - 管理端：
    - 信息检索与查看（运营视角）

## 阶段 4：收藏 / 举报 / 封禁与治理

**目标**：打造基础的内容治理闭环。

- **后端**
  - 收藏：`post_favorite` 增删查接口
  - 举报：`post_report` 提交与管理端处理（忽略/下架/封禁）
  - 用户封禁：封禁规则与封禁后行为（发布/拨号等受限）
- **前端**
  - 用户端：“收藏/取消收藏”“举报”入口
  - 管理端：举报列表、处理页，用户管理页（封禁 & 历史记录）

## 阶段 5：ES 检索与智能搜索（P1）

**目标**：先实现 ES 结构化搜索，再在此基础上加 AI 智能搜索。

- **后端**
  - ES 索引写入与同步（发布通过、更新、下架、过期）
  - 标准搜索接口：关键词 + 各类筛选 + 排序
  - AI 搜索：
    - Java `catering-ai` 调用 Python `catering-ai-service` 的解析接口
    - 将解析结果转换为 ES 查询条件
- **前端**
  - 用户端：
    - 搜索框 + 筛选结果列表
    - 智能搜索入口、失败 fallback 文案
  - 管理端：
    - 基于 ES 的信息检索工具（编号/电话/标题/城市）

## 前后端协作方式建议

- **按阶段 / 模块协同开发**：每一阶段切成若干“小模块”（如地区管理、审核队列、发布表单），由后端先定义接口草图（OpenAPI），前后端并行实现。
- **约定联调环境**：
  - 后端：`http://localhost:8080`
  - 管理端：`http://localhost:5173`（代理 `/api`）
  - 用户端 H5：`http://localhost:5174`（代理 `/api`）
- **接口文档**：通过 `/swagger-ui.html` 或 `/v3/api-docs/{group}` 作为单一事实来源。
