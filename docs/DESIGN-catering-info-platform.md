## Problem Statement

中国大量餐饮从业者（尤其 35–55 岁）依赖**微信群**获取招聘、转让、出租、求职等信息。信息分散、难检索、易过期、真假难辨，且无法按地区/岗位/薪资结构化查询。

目标：做一个**省级范围**的餐饮垂直信息平台，将微信群里的非结构化信息平台化，并通过**人工审核 + AI 检索**提升找信息效率。

### 核心用户角色

| 角色 | 典型诉求 | 关键行为 |
|------|----------|----------|
| 餐饮店主 | 招厨师/传菜/服务员；转店；出租店面 | 发布信息、筛选简历、联系候选人 |
| 餐饮从业者 | 找工作、看转让机会 | 浏览/搜索、投递、收藏 |
| 平台运营 | 保证信息质量、防诈骗 | 审核、下架、标注、数据统计 |

## Demand Evidence

**当前状态：假设阶段（Hypothesis）**

| 证据类型 | 状态 | 说明 |
|----------|------|------|
| 付费意愿 | ❌ 未验证 | 无用户愿意为发布/置顶付费 |
| 行为依赖 | ⚠️ 间接 | 微信群仍在用，但未验证会迁移到 App/小程序 |
| 具体用户 | ❌ 未命名 | 无具体店主/厨师姓名与访谈记录 |
| 运营触达 | ❌ 未验证 | 无已运营的微信群可导入冷启动 |

**Office Hours 判定：** 兴趣 ≠ 需求。在验证前，产品应设计为**最低成本验证假设**，而非一次性建全平台。

**建议的 2 周验证任务（The Assignment）：**
1. 加入或创建 **3 个**本地餐饮招聘/转让微信群（或向现有群主要到转发授权）。
2. 用 **飞书多维表格 / 腾讯文档** 手工结构化 50 条真实信息（招聘/转让/出租/求职各至少 10 条）。
3. 邀请 **5 名**店主 + **5 名**从业者试用「表格 + 简单 H5 列表页」搜索，记录：是否找到比翻群更快？是否愿意下次还来？

## Status Quo（用户现在怎么做）

```
微信群 → 长文本/图片海报 → 爬楼/搜聊天记录 → 电话/微信私聊 → 口头确认
```

**成本：**
- 信息 3–7 天过期，无人维护
- 无法按「某区 + 大厨 + 8000+」组合筛选
- 诈骗/虚假转让频发，靠熟人信用背书
- 年长用户不会用 58同城/BOSS 的餐饮垂直筛选

**真实竞品（非微信）：**
- 58同城：泛分类，餐饮不垂直，信息杂
- 天呈餐饮圈等：偏全国/加盟/供应链，招聘转让非唯一核心
- BOSS/智联：白领导向，厨师/传菜员覆盖弱
- 各类本地「餐饮交流群」：仍是微信生态

**你的机会：** 不是「第一个做招聘的」，而是**第一个把「某省餐饮圈微信群信息」结构化 + 可信 + 适老化**。

| **李师傅** | 38 岁，粤菜厨师，想换到市区大店 | 他会每天打开 App 还是只在急找工时打开？ |

> **行动项：** 下次 Office Hours 前，至少访谈 1 名真实「张老板」并记录原话。

### 关于「四类必须一起上」的前提挑战

你选择了四类同等重要。从 YC 视角，这是**平台愿景**，不是 **MVP 切口**。

**建议折中方案（统一发布，分阶段运营）：**

| 阶段 | 范围 | 说明 |
|------|------|------|
| **MVP** | 统一「信息发布」入口，四类都可选 | 技术上一体，**运营上只主推 1 类**（建议：招聘） |
| **V1** | 四类均衡曝光 + 审核 SLA | 招聘验证留存后再加大转让/出租 |
| **V2** | AI 对话式检索 | 依赖足够结构化数据量 |

**理由：** 四类共用「地区 + 联系方式 + 有效期 + 审核」底层，开发可一起做；但**冷启动内容**若四散，单类信息密度不够，AI 检索和列表体验都会空。

**推荐主推切口：门店招聘（厨师/传菜/服务员）**
- 频率最高（店主每月可能发）
- 从业者主动搜索意愿强
- 审核规则相对清晰（岗位/薪资/地点）
- 转让/出租低频但高客单价，适合第二阶段做「精品栏」

若你坚持四类同等推广，成功标准需改为：**上线 30 天内每类 ≥ 200 条有效信息**（省级），否则视为失败。

## Constraints

1. **用户年龄偏大** → 大字体、少步骤、微信登录、电话一键拨打、语音输入可选
2. **省级覆盖** → 必须强依赖「城市/区县」筛选；首页默认定位到用户所在市
3. **信息需审核** → 运营人力是瓶颈，需定义 SLA 与自动预审
4. **AI 找信息** → 依赖结构化字段 + 足够数据量；冷启动期 AI 是锦上添花
5. **当前无验证用户** → 第一版必须是「能验证假设的最小产品」

## Premises

请确认以下前提；若有异议，产品方向需调整。

| # | 前提 | 建议 |
|---|------|------|
| P1 | 目标用户愿意**离开微信群**来专门平台找信息 | 需验证；MVP 应支持「微信群分享链接回流」 |
| P2 | 省级范围在 MVP 阶段可行 | 可行，但**内容运营必须按城市切块**，不能省一级笼统运营 |
| P3 | 人工审核是信任核心 | 同意；可 AI 辅助预审，人工终审 |
| P4 | 四类信息可共享同一套发布/列表/搜索框架 | 同意；用「信息类型」字段区分 |
| P5 | 商业模式尚未定义 | MVP 免费；后续可考虑：置顶、会员、招聘成功佣金、转让中介认证 |

## Approaches Considered

### Approach A: 微信小程序 MVP（推荐）

**Summary：** 微信生态内完成登录、发布、浏览、分享；后端 + 简单管理后台。

| 维度 | 评估 |
|------|------|
| Effort | M |
| Risk | Low |
| Pros | 用户已在微信；分享回群零成本；适老化可做大按钮 |
| Cons | 小程序审核；复杂 AI 对话 UI 受限 |
| Reuses | 标准 CRUD + 审核工作流 |

### Approach B: H5 + 公众号（最快验证）

**Summary：** 纯 H5 列表/发布，公众号菜单入口；运营后台用现成 Admin 模板。

| 维度 | 评估 |
|------|------|
| Effort | S |
| Risk | Med |
| Pros | 1–2 周可上线；适合假设验证 |
| Cons | 体验弱于小程序；留存差 |
| Reuses | 与 A 同后端，可平滑升级 |

### Approach C: App + 全功能 AI 助手（理想架构）

**Summary：** 独立 App，AI 对话为主入口，智能匹配推送。

| 维度 | 评估 |
|------|------|
| Effort | XL |
| Risk | High |
| Pros | 体验上限高；推送、画像完整 |
| Cons | 年长用户安装门槛高；无验证需求前过度建设 |
| Reuses | 后端 API 可与 A 共享 |

**RECOMMENDATION：** **A（小程序 MVP）**，冷启动用 **B 的思维**（极简、快速验证），AI 先做「智能搜索框」而非独立 Chat 产品。

## Recommended Approach — 产品需求规格（PRD 摘要）

### 1. 信息类型与字段

#### 1.1 门店招聘

| 字段 | 必填 | 说明 |
|------|------|------|
| 岗位 | ✓ | 厨师/大厨/传菜/服务员/收银/店长/其他 |
| 城市、区县 | ✓ | 省级平台的核心筛选维度 |
| 门店类型 | | 中餐/火锅/烧烤/快餐/其他 |
| 薪资 | ✓ | 范围或面议；支持「包吃住」标签 |
| 招聘人数 | | |
| 工作要求 | | 年限、菜系、年龄偏好 |
| 联系人、电话/微信 | ✓ | 审核通过后展示 |
| 有效期 | ✓ | 默认 15 天，到期自动下架 |

#### 1.2 门店转让

| 字段 | 必填 | 说明 |
|------|------|------|
| 城市、区县、地址 | ✓ | 可模糊到路段 |
| 面积、租金 | ✓ | |
| 转让费 | ✓ | 面议/具体金额 |
| 经营类型、是否带设备 | ✓ | |
| 流水/盈利（可选） | | 敏感，审核重点 |
| 转让原因、图片 | | 最多 9 张 |
| 联系人 | ✓ | |

#### 1.3 店面出租

| 字段 | 必填 | 说明 |
|------|------|------|
| 城市、区县 | ✓ | |
| 面积、租金、转让费 | ✓ | |
| 是否可餐饮、明火 | ✓ | 餐饮特有 |
| 房东/中介 | | 需标注身份 |
| 图片、联系人 | ✓ | |

#### 1.4 求职

| 字段 | 必填 | 说明 |
|------|------|------|
| 期望岗位 | ✓ | |
| 期望城市 | ✓ | 可多选 |
| 工作年限、擅长菜系 | | |
| 期望薪资 | | |
| 个人简介 | | |
| 联系人 | ✓ | |

### 2. 功能模块

#### 2.1 用户端（小程序）

| 模块 | 功能 | 优先级 |
|------|------|--------|
| 首页 | 城市切换、四类入口、最新信息列表、搜索框 | P0 |
| 列表 | 按类型/地区/岗位/薪资筛选；排序：最新/即将过期 | P0 |
| 详情 | 结构化展示；一键拨号；收藏；举报 | P0 |
| 发布 | 选类型 → 填表 → 提交审核；草稿箱 | P0 |
| 我的 | 我的发布、收藏、浏览记录、消息（审核结果） | P0 |
| AI 搜索 | 自然语言：「XX 市 大厨 8000 以上」→ 解析为筛选条件 | P1 |
| 分享 | 生成卡片分享至微信群 | P0 |

**适老化要求：**
- 默认字号 ≥ 16px，关键按钮高度 ≥ 48px
- 发布流程 ≤ 4 屏
- 支持微信手机号一键登录
- 错误提示用口语化中文，避免技术术语

#### 2.2 运营后台

| 模块 | 功能 | 优先级 |
|------|------|--------|
| 审核队列 | 待审/通过/驳回；批量操作；驳回原因模板 | P0 |
| 信息治理 | 下架、标记虚假、重复检测 | P0 |
| 用户管理 | 封禁发布、举报处理 | P0 |
| 数据看板 | 按城市/类型的发布量、审核时效 | P1 |
| AI 预审 | 敏感词、明显虚假（如 0 元转让）、图片 OCR 辅助 | P2 |

**审核 SLA 建议：** 工作时段 2 小时内处理；超时自动提醒运营。

#### 2.3 AI 能力（分阶段）

| 阶段 | 能力 | 说明 |
|------|------|------|
| P1 | NL → 筛选 | 「帮我找省会附近的大厨招聘」→ city + job + salary |
| P2 | 相似推荐 | 看了 A 转让的用户推荐 B |
| P3 | 对话式顾问 | 「我想开火锅店，有没有合适铺面」— 需大量数据 |

**冷启动期：** AI 不是主入口；主入口仍是**分类 + 地区筛选**。

### 3. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 列表首屏 < 2s（4G） |
| 安全 | 手机号脱敏；防刷接口；发布频率限制 |
| 合规 | 用户协议；禁止虚假招聘/诈骗；实名可选（转让类建议强制） |
| 数据 | 信息默认 15–30 天过期；软删除保留审计 |

### 4. 技术架构（建议）

```
微信小程序
    ↓ HTTPS
API 服务（REST/GraphQL）
    ↓
PostgreSQL / MySQL — 用户、信息、审核记录
Redis — 缓存、限流
对象存储 — 图片
LLM API — AI 查询解析（P1）
管理后台 — React/Vue Admin
```

### 5. 冷启动与增长

1. **种子内容：** 与 5–10 个微信群合作，每日同步（需授权或机器人转发规则合规）
2. **地推：** 餐饮批发市场、厨具城、厨师学校周边
3. **激励：** 首发免费；前 100 条优质信息赠送「置顶 3 天」
4. **回流：** 每条信息分享卡片带小程序码，群内点击即回流

## Success Criteria

### MVP 成功（上线后 60 天）

| 指标 | 目标 |
|------|------|
| 有效信息量 | ≥ 500 条（全省），单城 ≥ 100 |
| 周活用户 | ≥ 200 |
| 审核时效 | 90% 在 4 小时内 |
| 留存 | 7 日留存 ≥ 15%（从业者） |
| 验证 | ≥ 3 个店主反馈「比翻群省时间」 |

### 失败信号（应 pivot）

- 30 天后单城有效信息 < 30 条
- 用户只愿看不愿发（供给不足）
- 审核跟不上，虚假信息导致用户流失
