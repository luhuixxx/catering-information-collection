## 1. 文档说明

### 1.1 目的

本文档描述「餐饮信息采集平台」的**产品需求**，供产品、设计、运营、研发对齐范围与验收标准。不涉及技术选型、接口与架构。

### 1.2 产品定位

面向**浙江省**的餐饮垂直信息平台，将微信群中分散的招聘、转让、出租、求职、招商加盟等信息**结构化上架**，经**人工审核**后供用户检索、联系；后期通过 **AI 辅助搜索**降低找信息成本。

### 1.3 产品边界

| 范围内 | 范围外（本期不做） |
|--------|-------------------|
| 五类信息的发布、浏览、搜索、分享 | 在线支付、合同、担保交易 |
| 人工审核与信息治理 | 全职 CRM、简历在线投递闭环 |
| 智能搜索（自然语言 → 筛选） | 独立 App（首期以微信生态为主） |
| 举报、下架、用户封禁 | 劳务中介入驻、食材供应链 |
| 运营免费置顶 | 置顶付费、会员体系（首期不做） |
| 适老化交互 | 多语言、海外市场 |

### 1.4 术语

| 术语 | 定义 |
|------|------|
| 信息帖 | 用户发布的一条招聘/转让/出租/求职记录 |
| 发布者 | 创建信息帖的用户（多为店主或求职者） |
| 浏览者 | 查看、搜索、收藏信息帖的用户 |
| 有效信息 | 审核通过且未过期、未下架的信息帖 |
| 审核 | 运营人员对信息帖合规性与真实性的人工判定 |
| 置顶 | 在列表前排展示，由运营后台免费设置 |
| 续期 | 信息快到期时，发布者一键延长展示时间，无需重新填表、无需再审 |

## 2. 背景与目标

### 2.1 问题背景

- 餐饮从业者（尤其 35–55 岁）习惯在**微信群**获取招聘、转店、铺面、求职信息。
- 信息以长文、海报图片为主，**难检索、易过期、真假难辨**。
- 通用招聘/房产平台对餐饮场景与年长用户不友好。

### 2.2 产品目标

| 目标 | 说明 | 衡量方式 |
|------|------|----------|
| 信息集中 | 省内餐饮相关信息可在一处查看 | 有效信息总量、城市覆盖数 |
| 找得更快 | 按地区、岗位、薪资等筛选，优于翻群 | 用户访谈、搜索成功率 |
| 更可信 | 人工审核 + 举报治理 | 虚假信息率、举报处理率 |
| 用得省心 | 大字号、少步骤、一键联系 | 发布完成率、适老化反馈 |

### 2.3 成功标准（上线后 60 天参考）

| 指标 | 目标 |
|------|------|
| 全省有效信息 | ≥ 500 条 |
| 单城有效信息 | ≥ 100 条（首批重点城市） |
| 周活跃用户 | ≥ 200 |
| 审核时效 | 90% 在工作时段 4 小时内完成 |
| 从业者 7 日留存 | ≥ 15% |
| 定性验证 | ≥ 3 名店主认为「比翻微信群省时间」 |

### 2.4 已确认的产品决策

| 事项 | 决策 |
|------|------|
| 服务区域 | **浙江省**，用户按省内地级市、区县筛选 |
| 电话展示 | **登录后才可看完整电话**；未登录仅浏览文字信息 |
| 续期 | **一键延长 15 天，免审核、不进审核队列**，每条最多续 1 次（见 5.3） |
| 地区数据 | **省/市/区县由管理端动态维护**，不写死在代码或前端配置里 |
| 招商加盟 | **允许发布**，作为第五类信息 |
| 置顶 | **免费**，仅运营后台手动设置 |
| 劳务中介 | **不做**中介入驻、资质认证等复杂角色 |
| 原则 | **流程从简**，不做多余认证和付费功能 |

## 3. 用户与场景

### 3.1 用户角色

#### 3.1.1 餐饮店主（发布方为主）

- **画像：** 中小餐饮店老板或店长，年龄偏大，微信熟练，不擅长用复杂 App。
- **诉求：** 快速招人；转让或出租店面；信息能被本地人看到。
- **权限：** 发布四类信息、管理自己的帖子、查看审核结果、分享至微信群。

#### 3.1.2 餐饮从业者（浏览方为主）

- **画像：** 厨师、传菜、服务员等，主动或被动找工作，偶尔关注转店机会。
- **诉求：** 按地区和岗位找活；电话联系；收藏感兴趣的信息。
- **权限：** 浏览、搜索、收藏、举报；可发布**求职**信息。

#### 3.1.3 平台运营人员

- **画像：** 内部运营/审核人员。
- **诉求：** 高效审核、处理举报、下架违规内容、查看数据。
- **权限：** 审核、下架、标注、封禁用户、配置驳回原因、查看统计（详见第 8 章）。

#### 3.1.4 平台管理员（可选，首期可与运营合并）

- **额外权限：** 运营账号管理、系统配置（城市列表、敏感词、有效期默认值等）。

### 3.2 核心用户故事

| ID | 角色 | 故事 | 验收要点 |
|----|------|------|----------|
| US-01 | 店主 | 我想发布一条服务员招聘，写明地点和薪资，审核通过后别人能打电话联系我 | 发布成功、审核流、详情展示电话 |
| US-02 | 从业者 | 我想在本市找大厨岗位，薪资 8000 以上 | 筛选/搜索准确、结果可拨号 |
| US-03 | 店主 | 我想把正在经营的店转让出去，附店面照片 | 支持多图、转让字段完整 |
| US-04 | 浏览者 | 我想找可做餐饮、有明火的铺面出租 | 出租帖筛选「可餐饮」「明火」 |
| US-05 | 求职者 | 我想发布求职信息，让老板联系我 | 求职帖发布与展示 |
| US-06 | 浏览者 | 看到可疑转让信息，我要举报 | 举报入口、运营可处理 |

| US-11 | 创业者 | 我想发布餐饮加盟招商信息 | 加盟类发布与展示 |

## 5. 信息帖生命周期

### 5.1 状态定义

| 状态 | 用户侧含义 | 是否公开展示 |
|------|------------|--------------|
| 草稿 | 未提交 | 否 |
| 待审核 | 已提交，等待运营 | 否 |
| 已通过 | 审核通过且在有效期内 | 是 |
| 已驳回 | 审核未通过 | 否（发布者可见原因） |
| 已过期 | 超过有效期 | 否（可展示为「已过期」占位或隐藏） |
| 已下架 | 运营或用户主动下架 | 否 |
| 已删除 | 用户删除（逻辑删除） | 否 |

### 5.2 状态流转

```
草稿 ──提交──► 待审核 ──通过──► 已通过 ──到期──► 已过期
                │                    │
                │驳回                ├──用户下架──► 已下架
                ▼                    ├──运营下架──► 已下架
              已驳回                  └──用户删除──► 已删除
                │
                └──修改后重新提交──► 待审核

已通过 ──一键续期──► 已通过（延长 15 天，免审核）
已过期 ──重新发布──► 待审核
```

### 5.3 业务规则

| 规则 | 说明 |
|------|------|
| 默认有效期 | 15 天；发布时可选 7 / 15 / 30 天 |
| 过期提醒 | 到期前 2 天提醒发布者，附「一键续期」入口 |
| 单用户发布频率 | 每自然日最多发布 5 条；超限提示明日再试 |
| 重复发布 | 24 小时内同电话+同类型+高度相似内容，提示可能重复 |
| **续期** | 见下方说明 |
| 重新发布 | 已过期/已驳回/已下架：编辑后重新提交，**需再审** |

**续期（产品说明）：**  
信息上架后有展示期限（默认 15 天）。到期后自动下架，不再出现在搜索结果里。  
**续期** = 发布者点击「延长 15 天」，帖子继续展示，**不用重新填表、不用再审**。每条帖子**最多续 1 次**；续期后仍过期，需走「重新发布」（编辑后提交审核）。这样避免微信群里过期信息没人管，同时也不给运营增加审核负担。

## 6. 用户端功能需求

以下描述**用户端**（首期以微信小程序形态交付，本文档只写行为与规则）。

### 6.1 注册与登录

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| AUTH-01 | 支持微信授权登录，首次登录即完成注册 | P0 |
| AUTH-02 | 获取用户昵称、头像用于「我的」展示（可拒绝，用默认头像） | P0 |
| AUTH-03 | 绑定手机号：发布信息前必须绑定手机号 | P0 |
| AUTH-04 | 未登录可浏览列表与详情，但**不可看电话**；点击拨号或看电话时引导登录 | P0 |
| AUTH-05 | 用户协议、隐私政策：首次发布前需勾选同意 | P0 |

### 6.2 首页

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| HOME-01 | 顶部展示当前城市，点击可切换（列表来自管理端已启用的城市） | P0 |
| HOME-02 | 首次进入根据定位推荐城市；定位失败则默认列表第一项或上次选择 | P0 |
| HOME-03 | 五个入口：招聘、转让、出租、求职、招商加盟（图标+文字，大点击区域） | P0 |
| HOME-04 | 全局搜索框：关键词搜索 + 进入高级筛选（P1：自然语言） | P0 |
| HOME-05 | 「最新信息」混合流或分 Tab，默认按发布时间倒序 | P0 |
| HOME-06 | 每条卡片展示：类型标签、标题/摘要、城市区县、关键字段、发布时间 | P0 |
| HOME-07 | 底部或悬浮「发布信息」主按钮 | P0 |

### 6.3 列表与筛选

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| LIST-01 | 按信息类型进入独立列表页 | P0 |
| LIST-02 | 筛选：城市、区县（联动）、各类型专属筛选项（见 4.2–4.5） | P0 |
| LIST-03 | 排序：最新发布（默认）、即将过期 | P0 |
| LIST-04 | 空状态：提示「暂无信息」+ 引导发布或切换城市 | P0 |
| LIST-05 | 分页或下拉加载，每页 20 条 | P0 |
| LIST-06 | 置顶帖在列表前部展示，带「置顶」标识（运营免费设置） | P1 |

### 6.4 搜索

| 需求 ID | 描述 | 优先级 |
|---------|------|--------|
| SRCH-01 | 关键词搜索：匹配标题、说明、岗位、地址等 | P0 |
| SRCH-02 | 搜索历史：本地保存最近 10 条 | P1 |

## 9. 内容安全与合规

### 9.1 发布禁止内容

- 虚假招聘（刷单、传销、缴纳费用上岗）。
- 虚假转让/出租/加盟（不存在店面、盗图、虚假品牌）。
- 涉黄涉暴、政治敏感、歧视性语言。
- 非餐饮相关广告（贷款、纯房产等）。
- **招商加盟允许发布**，但须为餐饮相关；夸大收益、保证回本类话术审核驳回。

### 9.2 审核原则

| 类型 | 重点核对 |
|------|----------|
| 招聘 | 薪资合理性、联系方式有效、无收费入职 |
| 转让 | 图片与描述一致、转让费/租金逻辑、防中介假盘 |
| 出租 | 可餐饮/明火与描述一致、身份标注 |
| 求职 | 无虚假履历、无买卖证件 |
| 招商加盟 | 品牌真实、无夸大回报、联系方式有效 |

### 9.3 用户协议要点（产品需体现）

- 发布者对信息真实性负责。
- 平台仅提供信息展示，不参与交易。
- 隐私：手机号展示规则、数据用于审核与治理。
- 违规可导致下架与封禁。

### 9.4 身份与认证（从简）

- 发布前绑定手机号即可，**不做**劳务中介入驻、资质认证、保证金等复杂流程。
- 转让/出租/加盟由运营审核把关，不单独建中介角色。

## 10. AI 能力需求（产品定义）

不涉及具体模型与部署，实现见 [TECH.md](./TECH.md) 第 5 章（Python LangChain + Java `catering-ai` 模块）。

### 10.1 智能搜索（P1）

**输入：** 用户自然语言，如「帮我找郑州金水区大厨，工资 8000 以上」。

**输出：**
- 解析出的筛选条件（可展示给用户确认）：城市、区县、岗位、薪资下限等。
- 符合的信息列表；无结果时给出放宽建议（扩大城市、降低薪资）。

**边界：**
- 仅搜索**已通过且未过期**的信息。
- 不支持闲聊；非搜索意图时引导使用筛选。

### 10.2 相似推荐（P2）

在详情页底部展示「看过的人还看了」最多 5 条，同城市、同类型、关键字段相近。

### 10.3 对话式顾问（P3，远期）

多轮对话，如「想开火锅店，有没有合适铺面」——需数据量充足后评估。
